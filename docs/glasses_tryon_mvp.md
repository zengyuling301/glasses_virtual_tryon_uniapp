# 眼镜试戴功能技术方案评估（MVP · 静态展示）

**文档性质**：技术可行性、实现路径、业务价值及工时粗估，供评审与排期对齐。  
**版本**：1.1  
**日期**：2026-05-12（初稿）  
**最近修订**：2026-05-13（对齐仓库实现进度，见 §9）

---

## 1. 背景与目标

针对眼镜试戴能力，双方从技术可行性、实现路径及业务价值等维度讨论后，确定 **以静态展示为主的 MVP**，核心产品目标为：

- **尺寸匹配预警**：通过拍照识别脸型/脸宽相关特征，推荐或排除不匹配的款式，而非单纯追求「美观展示」。
- **交互简化**：「拍照 → 选镜 → 展示」；重点告知「脸型与眼镜是否不匹配」，不承诺动态、毫米级物理贴合。

---

## 2. 已定技术决策摘要

| 决策项 | 结论 |
|--------|------|
| 高精度 3D 实时渲染 | **不采用**。手机前置景深信息不足、物理尺寸测量复杂度高，与 MVP 成本不匹配。 |
| 主实现路径 | **2D 位图叠加**：人脸关键点定位 → 归一化几何量 → 镜框 PNG 按关键点缩放/旋转/轻量形变 → 静态合成图。 |
| 开源 / 大模型 | **主路径不依赖** Hugging Face Diffusion 或 OpenAI 多模态 API；若需「炫技 Demo」可作**平行对比实验**（见 3.2）。 |
| Demo | 用开源人脸关键点方案快速搭建**静态 Demo**，验证方向与主观效果。 |

---

## 3. 技术实现路径

### 3.1 主路径：2D 叠加（推荐作为 MVP 核心）

1. **输入**：用户正面自拍（首版建议仅正面，降低姿态与标定难度）。
2. **检测**：开源方案如 **MediaPipe Face Landmarker**、**InsightFace** 等，输出人脸关键点或稠密网格。
3. **几何**：基于眼角、鼻翼、颧宽等计算**相对面宽**、**瞳距归一化比例**等，减弱拍摄距离带来的绝对像素误差。
4. **叠加**：每 SKU 提供**带 Alpha 的镜框 PNG**（MVP 一张正面即可），根据瞳距/鼻梁区域计算**缩放、平移、轻微旋转**；可选简单仿射/薄板样条等 **2D 形变**（仍非 3D 引擎）。
5. **输出**：静态合成图 + **可解释文案**（如：面宽与镜圈宽度不匹配，不推荐该款）。

**优点**：延迟可控、端侧可部署、商品轮廓可控、与「预警」业务一致。  
**局限**：侧面/大角度姿态效果有限；不宣称毫米级验光级精度。

### 3.2 备选：生成式「图片迁移」（可选 Demo）

- **含义**：Inpainting、ControlNet、IP-Adapter 等，将「佩戴眼镜」生成到人脸区域。
- **风险**：推理成本与延迟高、端侧难部署、镜框**商品还原度**与合规难控；「匹配/不匹配」结论难以**严格规则化**。
- **建议**：若需向客户展示「技术可能性」，可做**左右对比**：左列为规则叠加（正式 MVP），右列为生成式实验，并在文档中明确**商用以左为主**。

### 3.3 Demo 验证计划（可执行）

1. 准备 **50～100 张**授权测试人脸图、**约 10 款**镜框 PNG 及对应元数据样例。
2. 搭建离线脚本或小页面：**关键点 → 叠加 → 输出三联图**（原图 / 合成 / 匹配结论 UI）。
3. 记录：关键点失败率、大角度/遮挡 case、单张处理耗时；**不做**无标定条件下的毫米级真值对标。

**落地情况（截至 2026-05-13）**：第 2 点已在仓库内以 **极简 Web + CLI** 形式部分实现（上传/拍照 → 分析 → 选镜 → 单张合成图；匹配结论文案在 UI 展示）。第 1 点仍为 **少量 Demo 素材**（3 款线框 SKU + 元数据），尚未达到「50～100 张 / 约 10 款」规模；第 3 点的系统化记录与测试集整理**未做**。细则见 **§9 仓库实现进度**。

**参考实现**：`demo/try_on.py`（MediaPipe Face Landmarker + PNG 叠加）、`demo/app.py`（Flask Web MVP）、`demo/face_match.py`（分档与预警）、`assets/frames/catalog.json`（SKU 元数据）。用法见根目录 `README.md`。

## 4. 「眼镜匹配」专项方案概要

### 4.1 数据（SKU）

建议每款镜框至少维护：

- 镜圈宽度、鼻梁宽、镜腿长度、镜框几何总宽等；
- 可选：适用瞳距范围、脸宽分档（S/M/L）、性别/风格标签（业务用）。

素材：MVP **单张正面透明 PNG**；后续扩展非本文档范围。

### 4.2 算法与规则（可解释）

- **特征**：从关键点提取 3～5 个稳定相对量（如归一化脸宽、颞部宽度相对瞳距等）。
- **规则示例**：
  - 镜框总宽 vs 归一化脸宽：过窄 / 过宽 → **预警**；
  - 瞳距与镜架光学中心区域偏差过大 → **不推荐**或提示专业验光。
- **输出枚举建议**：`MATCH` | `WARN_TOO_WIDE` | `WARN_TOO_NARROW` | `LOW_CONFIDENCE`（附简短原因）。
- **低置信度**：检测分低、偏转过大、遮挡（口罩、墨镜）时，不输出强结论，引导重拍。

### 4.3 工程架构（示意）

- **端**：采集 + 可选本地推理（如 MediaPipe）或上传人脸图（需隐私与合规方案）。
- **服**：镜框元数据、可选服务端合成（便于水印、统计与 A/B）。

## 5. 业务价值与产品表述建议

- **对外**：强调「辅助选款与尺寸风险提示」，避免承诺「与线下试戴完全一致」。
- **对内**：成功指标可侧重「预警合理性」「用户重拍率下降」「选款转化率」等，而非单一主观美观分。

## 6. 经验工时（人天 · 粗估）

**假设**：1 名熟悉移动端或 Python 的视觉方向工程师为主力，产品/设计 0.5 人配合；镜框 PNG 与元数据初版由业务方提供。

| 模块 | 内容 | 经验人天 |
|------|------|----------|
| A | 人脸关键点集成、姿态/质量阈值、归一化度量 | 3～5 |
| B | 2D 叠加管线（缩放/旋转/轻量形变）、约 10 款镜框联调 | 3～5 |
| C | 匹配规则引擎、文案、低置信度分支 | 2～4 |
| D | 静态 Demo（脚本或极简 H5）：拍照 / 选镜 / 出图 | 2～4 |
| E | 隐私与合规初稿、异常 case 测试集整理 | 1～2 |
| **MVP 合计** | | **约 11～20 人天**（约 **2.5～4 周** 单人饱和） |

**可选增量**：HF/OpenAI 生成式对比 Demo，**+3～8 人天**（含选型、管线、API/GPU 与效果验收）。

**不含**：大规模设计改图、完整电商后台、法务深度审查；若扩展为**双端正式模块 + 埋点 + 审核**，整体可再乘 **1.5～2** 作粗算。

---

## 7. 风险与依赖

- **无距离标定**：难以输出绝对毫米；以**相对分档 + 预警**为主更稳妥。
- **素材质量**：PNG 透视、比例与真人自拍不一致时，叠加「违和感」上升，需设计规范。
- **合规**：人脸图像存储范围、保留周期、下载水印与告知同意需在方案阶段对齐法务。

---

## 8. 结论（一句话）

MVP 以 **2D 位图叠加 + 脸型/尺寸可解释规则** 实现「**不匹配预警 + 静态示意图**」；**不采用** 3D 实时与消费级手机景深做物理毫米测量；大模型路线仅作**可选对比 Demo**，不宜作为唯一匹配与商品还原依据。

## 9. 仓库实现进度（与 §6 模块粗对照）

以下为 **本仓库已落地能力** 与方案中模块 **A～E** 的对应关系，便于评审「方案 vs 代码」一眼对齐；**非**正式验收清单。

### 9.1 已完成（可用作技术验证 Demo）

| 能力 | 说明 | 对应方案章节 |
|------|------|----------------|
| 人脸关键点 | MediaPipe Face Landmarker（478 点），单人脸检测；虹膜环质心估瞳距，失败时退回内眦 | §3.1、§4.2 |
| 2D 叠加 | 镜框 PNG 按**眼内外眼角中点**定标定旋转 + **眼–鼻梁**混合定垂向 + Alpha 合成（`catalog` 仍用 `pupil_*_frac`/`eyeline_frac` 对齐镜片位）；CLI `demo/try_on.py`，`compose_try_on_bgr` 供 Web | §3.1 |
| 归一化面宽代理 | 颊部轮廓锚点（如 234/454）间距 ÷ 瞳距，得到无量纲比例；再分 **S / M / L** 三档（阈值可按业务样本再标定） | §3.1、§4.2 |
| 镜架元数据 | `assets/frames/catalog.json` + 透明 PNG；每 SKU 含 `band`、标称毫米宽、文案、镜片位 **`pupil_*_frac` / `eyeline_frac`**，可选 **`nose_bridge_blend`**（0～1，垂向眼线/鼻梁混合，默认见 `try_on.py`） | §4.1 |
| 可解释匹配状态 | `MATCH`、`WARN_TOO_NARROW`、`WARN_TOO_WIDE`、`LOW_CONFIDENCE`（附中文说明文案）；API 返回英文枚举，Web 界面展示中文标签 | §4.2 |
| Web MVP | `demo/app.py`：`/` 单页，`/api/analyze`、`/api/tryon`、`/api/catalog`；支持本地上传与 `capture` 拍照入口 | §3.3、§4.3 |
| 低置信度（初版） | 瞳距在图像中过小（像素阈值）时标为低置信并弱化结论；**未**接 blendshape 姿态/遮挡细分 | §4.2 |

### 9.2 部分完成 / 简化相对方案

| 项 | 方案预期 | 当前仓库 |
|----|-----------|----------|
| 镜框数量与素材 | 约 10 款商品级 PNG + 元数据 | **3 款** Demo 线框（`mvp_assets.py` 生成），非商品摄影 |
| 验证与数据 | 50～100 张授权人脸 + 指标记录 | **未建库**；仅依赖开发者自测与 OpenCV 示例图等 |
| Demo 形态 | 三联图（原图 / 合成 / 结论） | **单页流程**（预览 + 列表 + 出图），非固定三联导出 |
| 匹配规则深度 | 瞳距与镜架光学中心偏差等 | **未**单独实现「光学中心偏差」数值规则；以面宽分档 vs 镜架 `band` 为主 |
| 工程形态 | 端侧或正式服务端 + 合规 | **本地 Flask 开发服**，无持久化存人脸、无埋点/水印/法务流程 |

### 9.3 未做（仍属方案规划或可选增量）

- 生成式试戴对比（§3.2）与左右对比 UI。  
- 系统化的关键点失败率 / 大角度与遮挡 case 台账与耗时统计（§3.3 第 3 点）。  
- 隐私与合规初稿、异常 case 测试集整理（§6 模块 E）。  
- 薄板样条等更强 2D 形变、多 SKU 大规模设计联调（§3.1、§6 模块 B）。

### 9.4 后续建议（保持与 §7 风险一致）

1. **标定**：用真实授权样本统计颊宽/瞳距分位，重标 S/M/L 阈值，减少误判。  
2. **素材**：替换为商品级透明 PNG，并为每 SKU 校准瞳距锚点（与 `README.md` 中调参说明一致）。  
3. **合规**：若上线外网或留存用户图，需单独立项对齐告知同意、保留周期与水印等（§7）。
